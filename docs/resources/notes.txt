Part 3 — The precise division: determinism for the enumerable, agents for the ambiguous.
So the defensible position isn't "agents are better than playbooks." It's: the response space has an enumerable core and an ambiguous long tail. Determinism owns the core; agents own the tail. Mature SOAR is converging on exactly this hybrid — the 2026 buyer's-guide language describes "micro-agents scoped to narrow tasks coordinating autonomously and handing high-impact actions to humans." Pure-deterministic SOAR exists (it's the previous generation — Phantom, classic XSOAR playbooks) and its well-documented pain is precisely the long tail: rigid playbooks that can't handle novel or multi-step incidents and that over-fire, which is why the industry is moving to agentic.

Async FSM — what it is
FSM = Finite State Machine. An incident can only ever be in one of a fixed list of states, and it can only move between them via allowed transitions.

Think of it like a traffic light:

States: red → green → yellow → red
You can't jump from red directly to yellow
The light doesn't "decide" anything — it just follows the table
For Sentinel, an incident follows this path:


grounded → triaging → enriching → responding → resolved
                                              ↘ awaiting_approval
               ↘ escalated (anything goes wrong)
"Async" just means the code uses async/await — so while waiting for a triage result, the worker isn't frozen; it can handle other work.

Why not LangGraph? LangGraph is a framework for building graph-based agent workflows with checkpointing. It's powerful but heavy. 
Our incident's state is already saved in Postgres after every step — we don't need a separate checkpointer. 
The entire FSM is ~30 lines: a transition table + a while loop. LangGraph would be a dependency added for zero extra capability.
